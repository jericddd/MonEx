2. $MONEX payment — Option A (pay per purchase)

Status: Not implemented. Chosen approach: Option A — each shop buy = one on-chain payment. No in-game $MONEX balance (no deposit model).

Chain: Monad  |  First item: Monball packs in Shop tab  |  MVP architecture: Treasury wallet + server verification (A1).

What exists now (ready to extend)

  • Shop tab with gold / essence prices (SHOP_ITEMS in game)
  • X login + cloud save (MonExAuth, Cloudflare API)
  • No wallet connect, no on-chain verification yet

One-time setup flow

  1. Deploy or designate $MONEX token on Monad testnet.
  2. Create treasury wallet; set prices (e.g. 5 Monballs = 25 $MONEX).
  3. Add Connect Wallet + “Link to @handle” (sign message, no gas).
  4. Backend stores X user ↔ wallet address mapping.
  5. Backend can verify txs on Monad and reject reused hashes.

Every purchase flow (Option A1 — treasury)

  1. Player logged in with X → opens Shop → sees Monball pack priced in $MONEX.
  2. Connect wallet (Monad network). Link wallet if first time.
  3. Click Buy with $MONEX → wallet approves token (if needed) → sends $MONEX to treasury.
  4. Player pays gas in MON (native), price in $MONEX.
  5. Game sends txHash + itemId to API after confirmation.
  6. API verifies on-chain: correct chain, token, amount, recipient, sender = linked wallet, hash unused.
  7. API grants Monballs to cloud save → UI updates.

Security rules (non-negotiable)

  • Never grant items from client-only “payment succeeded” — always verify on server.
  • One transaction hash = one purchase (replay protection).
  • Payment wallet must match linked wallet for that X account.
  • Wait for block confirmations before granting.
