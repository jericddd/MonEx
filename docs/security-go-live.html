Security Go-Live

Plain-language checklist to keep monexmonad.xyz safe. You do not need to be a developer — follow the steps in order after merging security updates to main.

Already built into the game (code)

  • X login only on live — fake “dev login” is turned off in production API config.
  • No test cheat endpoints — simulate-mention and admin reset are disabled on live API.
  • Account-bound actions — syncing wild mons, pending catches, and personal logs require your login session.
  • Server-side save limits — gold, mons, gear, and progress are capped/validated on the server (anti-cheat).
  • Browser security headers — CSP, HSTS, anti-clickjacking, and related headers via _headers.
  • API rate limits — login, saves, sync, and admin routes are throttled per IP.
  • CORS lockdown — API only accepts browser calls from your site (and staging *.pages.dev).

Step 1 — Deploy the API (required)

Security fixes for login and saves live on the Cloudflare Worker, not just the website files.

  1. GitHub → repo MonEx → Actions
  2. Open Deploy Cloudflare API
  3. Click Run workflow → Run workflow
  4. Wait until the job shows green ✓

Step 2 — Cloudflare secrets (one-time / verify)

In Cloudflare dashboard → Workers & Pages → monex-api → Settings → Variables:

Secret / variableLive value
X_CLIENT_ID, X_CLIENT_SECRETSet (for real X login)
ENABLE_DEV_AUTH0 (or unset — repo default is 0)
ENABLE_SIMULATE0
ENABLE_ADMIN_RESET0
ADMIN_RESET_SECRETDelete on live (or leave unset)

Staging testers can temporarily set ENABLE_DEV_AUTH=1 in the Worker dashboard only — never on live.

Step 3 — Cloudflare WAF (recommended, free tier)

Cloudflare dashboard → your domain monexmonad.xyz → Security → WAF:

  • Turn on Bot Fight Mode (or Super Bot Fight Mode if available).
  • Add a Rate limiting rule for monex-api.*.workers.dev/api/* if you use a custom workers.dev hostname — e.g. 100 requests / minute / IP.
  • Enable Browser Integrity Check under Security → Settings.

Step 4 — GitHub branch protection

GitHub → repo → Settings → Branches → Add rule for main:

  • Require a pull request before merging
  • Do not allow force pushes
  • Optional: require status checks (staging build) before merge

This stops accidental or malicious direct pushes to the live site.

Step 5 — GitHub Actions secrets

Settings → Secrets and variables → Actions:

  • CLOUDFLARE_API_TOKEN — create with minimum permissions: Workers Scripts Edit, Workers KV Storage Edit, Account Settings Read.
  • CLOUDFLARE_ACCOUNT_ID — from Cloudflare sidebar.
  • Rotate the token if it was ever shared or committed.

Step 6 — Smoke test after deploy

  1. Open https://monexmonad.xyz/home.html in a private/incognito window.
  2. Confirm Login with X works (no dev handle prompt).
  3. Play briefly, refresh — progress should persist when logged in.
  4. Open dev tools → Network → confirm API calls go to monex-api.monexmonad.workers.dev (not an old dev URL).

Staging vs live

Staging (*.pages.dev) may use dev login if you enable it on the Worker — that is fine for testing. Live (monexmonad.xyz) must keep dev/simulate/admin flags off. Staging beta gold/gear cheats are hostname-gated in the game and do not affect live users.
